# Claude Skills 課程設計 - 14 小時完整教案

## 課程概述

**課程名稱：** Claude Skills 職場自動化課程  
**對象：** 非程式設計師 (行政、市場、HR、專案經理等)  
**總時數：** 14 小時  
**工具：** Claude Desktop  
**成果：** 學生建立 3-5 個自己工作上能用的 Skills，自動化重複任務  

---

## Module 1: Skills 基礎 (2 小時)

### 學習目標
- [ ] 理解什麼是 Skills 及為什麼需要
- [ ] 了解 SKILL.md 的基本結構
- [ ] 建立並測試第一個 Skill
- [ ] 在 Claude Desktop 中成功 invoke Skill

### 教學流程

#### 0:00-0:15 | 為什麼需要 Skills？
**講述方式：** 對比展示
- **沒有 Skill 的情況：** 用手動方式做一個任務 (例如：審查郵件)
- **有 Skill 的情況：** 同一個任務，用 Skill 做，展示節省時間 + 一致性

**核心訊息：**
> "Skills 是 AI 的『記憶』— 告訴 Claude 你做事的方式，它每次都按照你的標準來。"

#### 0:15-0:45 | SKILL.md 格式解析
**實時編輯一個簡單的 Skill**

```markdown
---
name: Writing style guide
description: 確保所有文件維持品牌一致的風格和語氣
---

# 寫作風格檢查

你是一個寫作教練，確保文件符合我們的風格標準。

當使用者分享草稿時，檢查：

1. **語氣**：是否專業且親切？(不僵硬，不隨便)
2. **清晰度**：非專家能理解每句話嗎？
3. **段落長度**：是否都在 150 字以下？句子在 20 字以下？
4. **專業術語**：有未解釋的行話嗎？應該移除或定義。
5. **結構**：有清楚的標題和列表嗎？

提供 3-5 個具體編修，解釋為什麼。用「修改前/修改後」展示重要改動。
```

**邊講邊做：** 逐行解釋每個部分
- `---` YAML frontmatter (名稱 + 說明)
- 空白行後是「指令」(告訴 Claude 怎麼做)
- 清單化、具體的要求

**💡 進階觀念：Skills 的行動力**
除了指令 (Instructions) 與資源 (Resources)，進階的 Skills 實際上是具備「行動力」的專家：
- **Function Calling (工具調用)**：教導 Claude 在何時、如何呼叫特定的工具。
- **MCP 整合**：可以指引 Claude 連結「本地 MCP Server」來執行檔案操作、讀取資料庫或存取內網。
- **程式碼執行**：搭配 Claude 內建的程式碼執行能力，在安全沙盒中進行數據運算與處理。

**🪄 捷徑：斜線指令 (Slash Commands)**
在 Skills 或 Plugins 中，您可以定義「斜線指令」（如 `/check`）。這能讓您在對話中快速調用特定功能，不需重新輸入 Prompt，是提升職場效率的極簡化操作。

#### 0:45-1:15 | 建立第一個 Skill (跟著做)
**學生在自己的 Claude Desktop 上建立**

步驟：
1. 開啟 Claude Desktop
2. Settings → Customize → Skills → Create
3. 複製上面的範本
4. 改成自己的工作內容 (例如：改成「郵件審查清單」)
5. 保存

**常見錯誤要即時指正：**
- ❌ "請檢查文法" → ✅ "列出 3 個文法錯誤並提供改正"
- ❌ "寫得更好" → ✅ "確保每個段落只有一個主要思想"

#### 1:15-1:45 | 測試 & 使用
**在真實情境下測試**

1. 貼一篇學生自己寫的文件到 Claude Desktop
2. 在聊天中輸入：`/[skill-name]`
3. 觀察 Claude 的反應
4. 討論：「結果對不對？」「怎麼改進 Skill？」

#### 1:45-2:00 | Q&A + 整理
- 學生提出他們遇到的問題
- 複習 Skill 的三個部分 (名稱、說明、指令)
- 預告下堂課會建立 3 個實用 Skill

#### 0:15-1:00 | 練習 1：郵件審查清單 (跟著做)

**情境：** 你經常寫郵件給客戶，想確保品質

**Skill 內容：**
```markdown
---
name: Email quality gate
description: 確保郵件在寄出前檢查關鍵要素
---

# 郵件品質檢查清單

評估郵件草稿，檢查以下內容：

## 必須檢查 (寄出前一定要過)
- [ ] 收件人明確嗎？
- [ ] 郵件回答了讀者的隱含問題嗎？
- [ ] 所有行動項目都有明確的期限嗎？

## 重要 (如果沒過，建議改)
- [ ] 開場的語氣符合你和收件人的關係嗎？
- [ ] 每個段落只有一個主要想法嗎？
- [ ] 沒有不必要的 CC/BCC 嗎？

## 加分項
- [ ] 拼寫和文法正確嗎？
- [ ] 簽名有嗎？

以「[項目名稱]: [具體改法]」格式反饋。
```

**教學步驟：**
1. 我先展示一個郵件例子，用 Skill 檢查
2. 指出什麼是「必須」、什麼是「加分」
3. 學生改自己的 Skill，試試看
4. 貼他們寫的郵件，讓 Claude 檢查
5. 看結果，調整 Skill 的要求

#### 1:00-1:45 | 練習 2：項目提案模板 (獨立做)

**情境：** 你需要向主管提案新項目

**Skill 骨架：**
```markdown
---
name: Project proposal formatter
description: 把你的想法轉成正式的項目提案
---

# 項目提案格式化

當使用者描述一個項目時，整理成以下結構：

## 要求是什麼
(一句話：我們要做什麼)

## 為什麼現在要做
(2-3 個重點：商業或營運的理由)

## 成功的樣子
(具體、可測量的成果)

## 需要的資源
(人員、時間、工具、預算)

## 風險 & 因應方案
(2-3 個關鍵風險 + 每個的解決方案)

## 時程
(里程碑 + 日期)

## 為什麼選這個方案
(考慮過哪些替代方案，為什麼這個最好)

輸出成可以直接複製到提案文件的 Markdown。
```

**學生要做的：**
1. 自己填入上面的模板
2. 用自己的一個項目想法測試
3. 看 Claude 的輸出，有沒有缺漏的地方
4. 調整 Skill 讓結果更完美

#### 1:45-2:30 | 練習 3：會議紀錄摘要 (獨立做)

**情境：** 會議後要整理重點給沒參加的人

**Skill 骨架：**
```markdown
---
name: Meeting summary generator
description: 把雜亂的會議紀錄轉成清晰的重點摘要
---

# 會議摘要產生器

從提供的會議紀錄中提取：

## 做了什麼決定

## 我要做的事
- [負責人] 要做什麼，[期限]
- [負責人] 要做什麼，[期限]

## 還不清楚的地方

## 下次會議應該討論什麼
- 基於本次的決定和遺留問題
```

## Module 3: 進階工作流 (3.5 小時)

### 學習目標
- [ ] 理解如何組合多個 Skills
- [ ] 設計一個 3-5 步的自動化流程

### 教學流程

#### 0:00-0:15 | 複習 + 導入
- 回顧 Module 2 的 3 個 Skill
- **新概念：** Skills 可以串連，一個的輸出是下一個的輸入

#### 0:15-1:45 | 範例 1：部落格內容創作管道

**完整流程展示：**

```
主題 → [Skill 1: 大綱生成] → 大綱
        ↓
     [Skill 2: 草稿寫手] → 初稿
        ↓
     [Skill 3: 編輯檢查] → 最終版本
```

**Skill 1: Blog outline generator**
```markdown
---
name: Blog outline generator
description: 從一個主題生成結構化的部落格大綱
---

# 部落格大綱產生器

根據主題：[使用者輸入]

生成一個包含以下的大綱：
- 開場 (讓讀者在乎的故事或問題)
- 3-4 個主要段落 (每個 300-400 字)
- 每個段落用「問題」或「承諾」作為標題
- 結尾 (回到開場，給讀者行動方向)

格式：Markdown H2 標題 + 下面用 bullet points
```

**Skill 2: Draft writer**
```markdown
---
name: Blog draft writer
description: 根據大綱寫出初稿
---

# 部落格草稿寫手

根據以下大綱寫 800-1000 字的初稿：

[使用者貼大綱]

寫作要求：
- 語氣親切但專業
- 開場要引人注目 (故事或統計)
- 用例子說明重點
- 結尾有明確的下一步

輸出：可直接用的初稿 Markdown
```

**Skill 3: Editor checklist**
```markdown
---
name: Blog editor checklist
description: 檢查部落格是否準備好發布
---

# 部落格編輯檢查

評估這篇部落格：

## 內容
- [ ] 標題清楚、吸引人嗎？
- [ ] 開場有 hook 嗎？
- [ ] 每個段落只有一個主要思想嗎？
- [ ] 例子具體且相關嗎？

## 風格
- [ ] 語氣一致嗎？
- [ ] 有未解釋的術語嗎？
- [ ] 段落長度合理嗎？

## 技術
- [ ] 拼寫和文法沒問題嗎？
- [ ] 連結都有效嗎？
- [ ] 格式一致嗎？

回饋格式：「修改前 → 修改後」
```

**教學步驟：**
1. 我選一個主題 (例如「如何有效開會」)
2. 現場示範完整流程：
   - 輸入主題到 Skill 1 → 得到大綱
   - 複製大綱，輸入 Skill 2 → 得到初稿
   - 複製初稿，輸入 Skill 3 → 得到編輯意見
   - 修改初稿，再跑一遍 Skill 3
3. 展示最終結果：從想法到發布品質的文章

**關鍵概念要強調：**
> "看到沒？第一個 Skill 的輸出，是第二個 Skill 的輸入。就像工廠的流水線一樣。"

```markdown
name: Customer tone detector
description: 識別客戶的情緒，指導回應語氣

根據客戶的信息，識別：
- **情緒狀態** (沮喪、困惑、高興、急迫)

輸出格式：「情緒: [emotion] | 關係: [type] | 問題: [issue]」

下一個 Skill 會用這個來選擇合適的回應模板。
```

```markdown
1. 誠摯道歉

3. 具體的解決方案
4. 預期的解決時間
5. 個人名字簽署

**如果情緒=困惑：**

**如果情緒=高興：**
[類似結構]

輸出：可直接寄出的郵件
```

**Skill 3: Response quality checker**
```markdown
---
name: Response quality checker
description: 在寄出前檢查客服回應
---

# 客服回應品質檢查

檢查這份客服回應：

## 內容
- [ ] 解決了客戶的實際問題嗎？
- [ ] 有明確的下一步嗎？
- [ ] 時間承諾現實嗎？
```

## Module 4: 職場自動化 (3.5 小時)

### 學習目標
- [ ] 識別工作中重複、耗時的任務
- [ ] 設計完整的自動化系統
- [ ] 測試並優化工作流
- [ ] 測量時間節省和品質改進

### 教學流程

#### 前 30 分鐘 | 問題識別

**每個學生回答：**
1. 你每週做什麼任務至少 2 次以上？
2. 每次要花多久？
3. 為什麼必須做這個？ (目的是什麼)
4. 有什麼限制或規則？

**我幫助他們選最有潛力的 (省時間 + 重複)：**
- ✅ 週報生成 (每週、重複、規則明確)
- ✅ 文件填寫 (常常做、規則多、變數少)
- ✅ 會議準備 (週期性、結構化)
- ❌ 需要創意決定的工作 (不適合自動化)

#### 後 3 小時 | 三個實作案例

**案例 1：週報生成系統 (1 小時)**

**流程設計：**
```
週一-週五記錄
   ↓
[Skill 1: 資料提取] → 分類成成果、blockers、洞察
   ↓
[Skill 2: 分析] → 找出趨勢、風險、機會
   ↓
[Skill 3: 格式化] → CEO-ready 的週報
   ↓
[Skill 4: 品質檢查] → 檢查完整性、語氣、數字
```

**每個 Skill 的大綱：**
- Skill 1: 從筆記中抽出「完成了什麼」、「遇到什麼問題」、「下週計畫」
- Skill 2: 分析週的成果，找出 1-2 個關鍵洞察
- Skill 3: 用公司的格式 (標題、表格、重點標示) 排版
- Skill 4: 檢查有沒有 KPI、有沒有遺漏的專案、語氣一致

**實作步驟：**
1. 我展示一個完整的週報案例 (從記錄 → 最終版本，每一步展示)
2. 學生提供他們的週報，我們一起試試
3. 討論：用 AI 前要 30 min，用 AI 後要多久？品質如何？

**目標成果：**
```
[本週週報]

成果：
- 完成 A 專案的初稿審查
- 帶領 B 會議，確認了方向
- 寫完 C 文件

Blockers：
- D 的回饋還沒來，等待中 (預期 [date])

下週計畫：
- 根據 D 的回饋修改 C
- 開始 E 專案的規劃
- 參加 F 會議

關鍵洞察：
- A 專案預計比原訂早 2 週完成
- 發現了 D 過程中的一個風險，已向主管報告
```

**案例 2：文件管理系統 (1 小時)**

**情境：** 常需要客製化文件 (合約、提案、報告)

**流程：**
```
客戶資訊 (名字、類型、預算等)
   ↓
[Skill 1: 模板選擇器] → 決定用哪個模板
   ↓
[Skill 2: 資料填充] → 自動填入變數
   ↓
[Skill 3: 品牌檢查] → 確保格式、logo、色系對
   ↓
[Skill 4: 法務檢查] → 如適用，檢查法律語言
   ↓
最終文件，可直接列印或寄出
```

**實作步驟：**
1. 提供 2-3 個 Skill 範本 (客戶提案、員工合約、會議紀錄)
2. 學生選一個他們真的需要
3. 試著客製化，填入他們的規則
4. 用真實資料測試

**案例 3：會議準備系統 (1 小時)**

**流程：**
```
會議清單 (日期、與會者、目的)
   ↓
[Skill 1: 背景蒐集] → 蒐集相關最新信息
   ↓
[Skill 2: 議程生成] → 根據與會者和目的設計議程
   ↓
[Skill 3: 資料準備] → 準備會議所需的數據、圖表
   ↓
[Skill 4: 郵件撰寫] → 寄給與會者，包含議程和附件
   ↓
完整的會議包，與會者一收到就知道要準備什麼
```

### 2. 常見錯誤 & 修正

| 錯誤 | 為什麼不行 | 怎麼改 |
|------|---------|------|
| "檢查拼寫" | 太模糊，AI 不知道標準 | "檢查 3 個最常見的拼寫錯誤，列出來" |
| "寫得更好" | 「更好」的定義模糊 | "改進的標準是：短句、主動語態、具體例子" |
| "做 X 和 Y" | 大部分一個 Skill 只做一件事 | 分成兩個 Skill，或明確說「然後寄給誰」 |
| Skill 太長 | 指令超過 300 字，AI 容易遺漏 | 分成多個 Skills，或只寫最重要的 5 點 |

### 3. 學生檢查清單

**建立 Skill 時：**
- [ ] 名稱清楚嗎？(讀了就知道做什麼)
- [ ] 說明是一句話嗎？(簡潔)
- [ ] 指令清楚嗎？(有例子)
- [ ] 要求具體嗎？(不是「更好」，而是「短於 50 字」)
- [ ] 試過嗎？(用真實內容)

**使用 Skill 時：**
- [ ] 輸入清楚嗎？(AI 知道你要什麼)
- [ ] 結果符合預期嗎？
- [ ] 如果不符，Skill 要改哪裡？

- ✅ "試試把『更好的寫法』改成『用主動語態和短句』，這樣 AI 更清楚。"
- ❌ "這裡寫得不清楚。"
